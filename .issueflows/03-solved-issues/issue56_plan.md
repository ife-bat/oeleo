# Issue #56 — Plan: Create proper documentation

## Goal

Stand up a Zensical-based docs site under `docs/`, verify it builds/serves locally, and add a root `.readthedocs.yaml` so @jepegit can wire Read the Docs without further repo plumbing.

## Constraints

- Docs toolchain is **Zensical** (not Sphinx/MkDocs as primary). Prefer native `zensical.toml` over transitional `mkdocs.yml`.
- Publish path is **Read the Docs**; do not add a competing GitHub Pages workflow.
- RTD *project* registration stays human-owned (`@jepegit`); this issue only ships buildable config + content.
- Keep day-to-day app/runtime deps lean — Zensical is a docs tool, not a runtime dependency of `oeleo`.
- Stay on `uv` / `pyproject.toml` conventions ([`this-project.md`](../04-designs-and-guides/this-project.md), DOC-01 in [`code-review-packaging-and-ops.md`](../04-designs-and-guides/code-review-packaging-and-ops.md)).
- No version bump required for docs-only work unless we also change published package metadata in a release-worthy way.
- `.gitignore` already ignores `/site` (Zensical default `site_dir`) — keep build output untracked.

### Prior art

- **Toolbox:** none (`00-tools/` empty).
- **Graph:** Community 4 holds much of the current user-facing README content (install, `.env`, examples); Community 8/0 are Worker/DB concepts useful as doc topics. No existing docs-site nodes.
- **Repo:** rich [`README.md`](../../README.md) is today’s docs surface; no `docs/`, `zensical.toml`, or `.readthedocs.yaml` yet. `.gitignore` already has `/site` and `docs/_build/`.
- **External:** [Zensical new project](https://zensical.org/docs/usage/new/) → `zensical.toml` + `docs/`; [RTD + Zensical](https://docs.readthedocs.com/platform/stable/intro/zensical.html) → `build.jobs` install/build + copy `site/` into `$READTHEDOCS_OUTPUT/html/`.
- **Convention:** hand-author Markdown from README rather than inventing a parallel narrative; do **not** introduce mkdocstrings/API autodoc in this issue (can coexist later).

## Approach

1. **Scaffold (hand-craft, not `zensical new`)** so we skip the bundled GitHub Pages workflow and avoid clobbering the existing repo layout.
   - Root [`zensical.toml`](../../zensical.toml): `site_name`, `repo_url`/`repo_name`, `docs_dir = "docs"`, `site_dir = "site"`, explicit `nav`, optional placeholder `site_url` once the RTD slug is known (can land empty / omitted initially).
   - `docs/` Markdown pages split from README into a small nav:
     - `index.md` — what oeleo is, features/limitations, quick links
     - `install.md` — PyPI + uv/dev install
     - `usage.md` — Worker flow, local example, scheduler, SSH example
     - `configuration.md` — `.env` sample + env-var reference (core / SSH / SharePoint / app)
     - `database.md` — `filelist` schema + `code` meanings
     - `development.md` — uv sync/test/format/build (from README Development)
   - Drop or fold “Status / Future / Hints” into `index.md` lightly; no need for a separate page each.

2. **Tooling**
   - Add a `[dependency-groups] docs = ["zensical"]` entry in `pyproject.toml` (or pin a recent version) so local work is `uv sync --group docs` then `uv run zensical serve` / `uv run zensical build`.
   - Document those two commands in `docs/development.md` and a short README pointer.

3. **Read the Docs config**
   - Add root [`.readthedocs.yaml`](../../.readthedocs.yaml) following the official Zensical recipe: Ubuntu + Python, `pip install zensical` (or a tiny pinned `docs/requirements.txt` for reproducible RTD builds), `zensical build`, copy `site/*` → `$READTHEDOCS_OUTPUT/html/`.
   - Prefer a pinned `docs/requirements.txt` referenced from the YAML if locking the RTD install is cheap; otherwise unpinned `pip install zensical` is acceptable for v1.

4. **README slim-down**
   - Keep install blurb + one minimal example + link to `docs/` (and later the RTD URL).
   - Avoid duplicating the full env-var / DB tables once they live under `docs/`.

5. **Verify locally (during `/iflow-start`, not here)**
   - `uv run zensical build` must succeed.
   - Spot-check with `uv run zensical serve` that nav and pages render.

6. **Out of scope**
   - Creating/linking the RTD project in the RTD UI.
   - GitHub Pages / `.github/workflows/docs.yml`.
   - Autogenerated API reference, tutorials beyond README migration, i18n, custom theme deep-dives.
